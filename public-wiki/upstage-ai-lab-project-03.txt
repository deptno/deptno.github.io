10:T2b19,<h1 id="mlops-프로젝트">mlops 프로젝트</h1>
<h2 id="개요">개요</h2>
<ul>
<li>과목: mlops</li>
<li>주제: 영화 추천 시스템</li>
<li>modeling 쪽으로 공부도 할 겸 할까 하였으나 infra 처리 가능 인력이 없는 것 같아서 ops 영역을 처리<ul>
<li>개인적으로도 이것 저것 시간 뜯기는 것들이 있어서 여유있게 익숙하면서도 배울 것이 있는 분야 선택</li>
<li>프로젝트상으로도 ops를 main goal 로 인식, 팀원들의 인식과는 차이가 있는 느낌</li>
</ul>
</li>
<li>수업은 다 듣고싶었으나 이전에 들었던과 중복이 많고 프로젝트에 집중이 필요하여 많이 못들음<ul>
<li>억지로 듣는것보다 손으로 하는게 더 남을 것 같은 느낌</li>
</ul>
</li>
<li>personal goal:<ul>
<li>뭐라도 유용한 새로운 것 경험</li>
<li><input checked="" disabled="" type="checkbox"> mlops 를 경험<ul>
<li><input checked="" disabled="" type="checkbox"> <a href="mlflow">mlflow</a> 와 <a href="airflow">airflow</a> 를 통해 modeling 부터 deploy 까지 전 cycle을 구현한다</li>
<li><input checked="" disabled="" type="checkbox"> <a href="mlflow">mlflow</a><ul>
<li><input checked="" disabled="" type="checkbox"> experiment 를 통해 log 심기</li>
<li><input checked="" disabled="" type="checkbox"> model registry 를 활용하여 모델을 관리한다<ul>
<li><input checked="" disabled="" type="checkbox"> api 를 통한 동적 tag 제어<ul>
<li><input checked="" disabled="" type="checkbox"> 동적으로 <em>production</em> 버전을 관리한다</li>
</ul>
</li>
</ul>
</li>
</ul>
</li>
</ul>
</li>
<li><input checked="" disabled="" type="checkbox"> <a href="docker-compose">docker-compose</a> 로 개발 환경을 <em>local only</em> 구성<ul>
<li>이제까지는 <a href="kubernetes">kubernetes</a> 홈 서버를 가지고 있기 때문에 <a href="docker-compose">docker-compose</a> 를 그닥 활용하지 않았음</li>
<li><input checked="" disabled="" type="checkbox"> 이참에 <a href="docker-compose">docker-compose</a> 를 활용<ul>
<li><input checked="" disabled="" type="checkbox"> 팀원마다 개발에 대한 지식이 다르고 설치 자체가 장벽인 경우도 있음</li>
<li><input checked="" disabled="" type="checkbox"> 개발인력으로 구성된 프로젝트가 아님<ul>
<li><input checked="" disabled="" type="checkbox"> <a href="docker-compose">docker-compose</a> <code class="span">up</code> 명령 하나로 환경 구축 목표</li>
</ul>
</li>
</ul>
</li>
</ul>
</li>
<li><input checked="" disabled="" type="checkbox"> <a href="fastapi">fastapi</a><ul>
<li><input checked="" disabled="" type="checkbox"> <a href="python">python</a> 의 가벼운 backend framework 경험</li>
<li><input checked="" disabled="" type="checkbox"> 재시작없는 동적 model update 지원</li>
</ul>
</li>
<li><input disabled="" type="checkbox"> 시간 남으면 팀원들이 작성한 modeling 검토</li>
<li><input checked="" disabled="" type="checkbox"> ui 개발은 피한다</li>
<li><input checked="" disabled="" type="checkbox"> crawling 은 피한다</li>

<h2 id="문제">문제</h2>
<ul>
<li><a href="airflow">airflow</a> <del>문제</del><ul>
<li><a href="kubernetes">kubernetes</a> 환경이 아니다 보니 <a href="pod">pod</a> 로 독립된 컨테이너를 올릴 수 없어 DAG 을 실행하는 <a href="python">python</a> script 의 모든 라이브러리를 담아야했음<ul>
<li>이를 위해서 <a href="airflow">airflow</a> image + 사용 라이브러리를 담아 커스터 빌드해야 했음</li>
<li>실제 환경에서는 <a href="kubernetes">kubernetes</a> 관련 operator 가 사용될 것이므로 이는 <strong>프로젝트 환경 특수</strong></li>
</ul>
</li>
</ul>
</li>
<li>api server 에서 모델의 <code class="span">pkl</code> 파일을 불러올때 DAG 에 사용된 <a href="python">python</a> 파일에 종속이 걸려는 문제<ul>
<li><a href="mlflow">mlflow</a> 는 주류 라이브러리에 대해서 autolog 등 편의성을 제공</li>
<li>사용 모델이 <code class="span">implicit</code> 라이브러리를 사용하는 모델이라 비주류</li>
<li><a href="/public-wiki/diary/2024-10-11">[diary] diary:2024-10-11</a> 시점 기준 <code class="span">mlflow.pyfunc.log_model</code> 함수는 wrapper 를 요구<ul>
<li>때문에 wrapper 관련된 코드가 저장된 모델을 불러오는 쪽에서도 필요<ul>
<li>해당 위치는 api server 쪽</li>
</ul>
</li>
<li><input checked="" disabled="" type="checkbox"> code sharing 의 우아한 처리 고민은 프로젝트 여건상 아닌 것 같아 <a href="airflow">airflow</a> 커스텀 이미지 빌드시에 관련 코드 함께 말아버림</li>
<li>아래는 이해를 돕기 위한 코드<pre><code class="hljs language-python"><span class="hljs-keyword">class</span> <span class="hljs-title class_">ALSWrapper</span>(mlflow.pyfunc.PythonModel):

- flow 의 시작인 [[airflow]] trigger 가 daily trigger 라 **가정** 하고 수동 트리거
- [[airflow]]
  - DAG 실행
    - `TASK1:`
      - 순서에 맞는 dataset 을 가져옴
      - model(ALS) 생성
        - [[mlflow]] 에 parameter 및 model 저장
          - 최신 model 을 `production` tagging, 나머지는 `archived` tagging
    - `TASK2:`
      - `production` tag 가 붙은 방금 생성된 model 을 api 서버에 전달
- [[fastapi]] 서버
  - `TASK2` 의 request 를 받아 새로운 model 로 서빙중인 모델을 변경
  - 서버는 계속 모델을 서빙중

## 문제
- [[airflow]] ~~문제~~
  - [[kubernetes]] 환경이 아니다 보니 [[pod]] 로 독립된 컨테이너를 올릴 수 없어 DAG 을 실행하는 [[python]] script 의 모든 라이브러리를 담아야했음
    - 이를 위해서 [[airflow]] image + 사용 라이브러리를 담아 커스터 빌드해야 했음
    - 실제 환경에서는 [[kubernetes]] 관련 operator 가 사용될 것이므로 이는 **프로젝트 환경 특수**
- api server 에서 모델의 `pkl` 파일을 불러올때 DAG 에 사용된 [[python]] 파일에 종속이 걸려는 문제
  - [[mlflow]] 는 주류 라이브러리에 대해서 autolog 등 편의성을 제공
  - 사용 모델이 `implicit` 라이브러리를 사용하는 모델이라 비주류
  - [[diary:2024-10-11]] 시점 기준 `mlflow.pyfunc.log_model` 함수는 wrapper 를 요구
    - 때문에 wrapper 관련된 코드가 저장된 모델을 불러오는 쪽에서도 필요
      - 해당 위치는 api server 쪽
    - [X] code sharing 의 우아한 처리 고민은 프로젝트 여건상 아닌 것 같아 [[airflow]] 커스텀 이미지 빌드시에 관련 코드 함께 말아버림
    - 아래는 이해를 돕기 위한 코드
      ```python
      class ALSWrapper(mlflow.pyfunc.PythonModel):
          def __init__(self, model):
              self.model = model

          def load_context(self, context):
              pass

          def predict(self, context, model_input):
              return self.model.recommend(model_input)

      mlflow.pyfunc.log_model(
          f'{experiment_name}',
          # model 은 훈련을 마친 model
          python_model=ALSWrapper(model)
      )
      ```

  - [[mlflow]] 는 주류 라이브러리에 대해서 autolog 등 편의성을 제공
  - 사용 모델이 `implicit` 라이브러리를 사용하는 모델이라 비주류
  - [[diary:2024-10-11]] 시점 기준 `mlflow.pyfunc.log_model` 함수는 wrapper 를 요구
    - 때문에 wrapper 관련된 코드가 저장된 모델을 불러오는 쪽에서도 필요
      - 해당 위치는 api server 쪽
    - [X] code sharing 의 우아한 처리 고민은 프로젝트 여건상 아닌 것 같아 [[airflow]] 커스텀 이미지 빌드시에 관련 코드 함께 말아버림
    - 아래는 이해를 돕기 위한 코드
      ```python
      class ALSWrapper(mlflow.pyfunc.PythonModel):
          def __init__(self, model):
              self.model = model

          def load_context(self, context):
              pass

          def predict(self, context, model_input):
              return self.model.recommend(model_input)

      mlflow.pyfunc.log_model(
          f'{experiment_name}',
          # model 은 훈련을 마친 model
          python_model=ALSWrapper(model)
      )
      ```
