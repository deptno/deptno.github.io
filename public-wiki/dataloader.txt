10:T726,<h1 id="dataloader">dataloader</h1>
<h2 id="코어-피쳐">코어 피쳐</h2>
<ul>
<li>배치<ul>
<li>각각의 요청을 모아서 nexttick 타이밍에, 배치 요청을 함수를 호출하여 한번에 데이터를 가져온다</li>
</ul>
</li>
<li>캐싱<ul>
<li>요청에 대한 값을 캐싱하여 디비에 두번 쿼리하지 않는다</li>
</ul>
</li>
</ul>
<h2 id="사용을-위해-라이브러리">사용을 위해 라이브러리</h2>
<ul>
<li><strong>깃허브</strong> <a href="https://github.com/deptno/dataloader-toolbox" target="_blank">[URL] https://github.com/deptno/dataloader-toolbox</a></li>
</ul>
<h3 id="설명">설명</h3>
<blockquote><p>사용했던 dataloader-toolbox 코드를 기준으로 의미를 역추적하여 그럴것이다라는 추론으로 빠르게 정리</p></blockquote><ul>
<li>요청당 dataloader session 을 생성</li>
<li>session 은 <code class="span">WeakMap</code> 형태, function 주소를 key 값으로 가짐</li>
<li>database query 요청 함수에 대해서 dataloader <strong>wrapper</strong> 를 생성하고 session 에 저장<ul>
<li>original 함수는 배치 처리를 위한 <code class="span">T[]</code> 인자를 받는 함수</li>
<li><strong>wrapper</strong> 는 <code class="span">.load</code> 메소드를 통해서 <code class="span">T</code> 를 인자로 받는 함수</li>
</ul>
</li>
<li>해당 함수를 resolver 의 여러 곳에서 호출하게될 수 있는데 이때 마다 wrapper 를 획득하여 호출(캐싱처리)</li>
<li>해당 요청이 모여 nexttick 에 배치 처리를 하게되면서 실제 함수인 original 함수가 호출</li>
<li>응답이 각 <strong>wrapper</strong> 의 호출부였던 <code class="span">.load(T)</code> 에 응답 분배됨</li>
</ul>
